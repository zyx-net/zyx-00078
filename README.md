# 社区团购售后仲裁系统

面向团长、商家、客服三类角色的售后仲裁管理系统，支持售后申请登记、状态受控流转、版本审计追踪和数据持久化。

## 四、核心功能

### 1. 三类角色
- **团长**：发起售后申请（缺货/破损/错发）、补充凭证
- **商家**：查看售后、响应处理、上传商家凭证
- **客服**：仲裁裁决（同意退款/驳回）、导出退款清单

### 2. 售后类型队列
- 缺货（outOfStock）
- 破损（damaged）
- 错发（wrongDelivery）

### 3. 状态流转（受控状态机）
```
待举证(pendingEvidence)
    ↓ 团长提交凭证
商家处理(merchantProcessing)
    ↓ 商家响应
客服仲裁(csArbitration)
    ↓ 客服裁决           ↓ 客服驳回
退款完成(refundCompleted)  驳回(rejected)
```

### 4. 版本控制
- 每步操作递增版本号
- 使用乐观锁防止旧版本重复处理
- 完整审计历史：操作人、备注、时间、版本号

### 5. 筛选查询
- 按售后类型筛选
- 按状态筛选
- 按责任方筛选
- 关键词搜索

### 6. 数据持久化
- 使用SQLite本地文件存储
- 重启后队列顺序、审计历史、案件状态、导出内容完全一致

## 六、批量退款确认功能

### 功能概述
客服可以从已进入「客服仲裁」状态的案件中勾选多笔，批量同意退款或批量驳回。系统提供乐观锁校验、部分失败处理、审计追踪和CSV导出能力。商家和团长无权限使用此功能。

### 核心特性
- **权限控制**：仅客服（cs角色）可见批量操作入口和调用API
- **预览校验**：提交前展示每笔当前状态、版本号、退款金额汇总和不可处理原因
- **乐观锁**：逐笔使用版本号校验，已被他人处理的案件不会被误改
- **部分失败**：返回详细的成功/失败/跳过明细，不影响其他正常案件
- **审计追踪**：每笔批量操作生成批次号，完整记录操作人、备注、每笔结果
- **CSV导出**：支持按批次导出操作结果
- **数据持久化**：重启后批次记录、案件状态、审计和导出内容完全一致

## 七、撤销最近批次功能

### 功能概述
客服在批量操作历史里能对自己刚执行且仍未被后续处理的批次发起撤销，撤销前展示每笔案件原状态、新状态、当前版本和不可撤销原因；确认后逐笔用乐观锁回滚到批次前状态，已经被单独处理或再次批量处理过的案件只记失败明细，不影响其他可撤销项。团长、商家不能看到入口或调用接口，客服也不能撤销别人的批次。撤销动作要写入审计历史和批次详情，能导出CSV，重启后撤销记录、案件状态、版本号和导出内容保持一致。

### 核心特性
- **权限控制**：仅客服（cs角色）可见撤销入口和调用API，且只能撤销自己创建的批次
- **撤销预览**：提交前展示每笔案件的当前状态、当前版本、批次后状态、批次后版本、撤销目标状态、撤销目标版本，以及不可撤销原因
- **乐观锁回滚**：逐笔使用版本号校验，已被他人处理的案件不会被误改
- **部分失败处理**：返回详细的成功/失败/跳过明细，不影响其他正常案件
- **审计追踪**：每笔撤销操作生成审计记录，完整记录撤销人、备注、每笔结果
- **双重记录**：撤销信息同时写入批次表、批次明细表、撤销审计表、撤销明细表
- **CSV导出**：支持按批次导出包含撤销信息的操作结果，支持按撤销记录导出撤销明细
- **数据持久化**：重启后撤销记录、案件状态、版本号和导出内容完全一致

### 不可撤销原因说明
| 原因 | 说明 |
|------|------|
| 案件已被后续处理 | 案件版本号与批次处理后版本不一致，说明案件已被单独处理 |
| 案件状态已变更 | 案件状态与批次处理后状态不一致 |
| 案件已被其他批次处理 | 该案件已被其他后续批次成功处理 |
| 案件在批次中未成功处理 | 批次中该案件处理状态不是成功，无需撤销 |

---

**前置条件**：
- 已完成一笔批量退款操作，所有案件处理成功
- 这些案件未被后续单独处理或再次批量处理

**操作步骤**：
1. 登录客服账号 `cs1` / `123456`
2. 点击左侧「批量操作历史」
3. 在列表中找到刚创建的批次，确认：
   - ✅ 批次号后没有「已撤销」标签

4. 点击「撤销」按钮，打开撤销弹窗：

   - ✅ 三步流程指示器显示当前在「预览」步骤

   - ✅ 每笔案件显示：当前状态、当前版本、批次后状态、批次后版本、撤销目标状态、撤销目标版本

   - ✅ 所有案件的「撤销目标版本」为批次处理前的版本号（如v3）

## 八、售后仲裁规则配置与自动分派功能

### 功能概述
客服可以维护按售后类型、责任方、退款金额区间和商家维度匹配的仲裁规则，规则能设置默认处理建议、优先级、启停状态和备注。团长提交案件或商家响应后，系统自动给出命中的规则、建议动作和分派客服，案件详情里能看到命中原因。规则列表支持导入/导出CSV，导入时处理重复优先级、无效区间和并发版本冲突。只允许客服管理规则，团长和商家只能看到案件上的建议结果。规则变更、自动命中、人工覆盖都要写审计日志，SQLite重启后配置、命中记录、分派结果和导出内容保持一致。

### 核心特性
- **多维度匹配**：支持按售后类型、责任方、退款金额区间、商家ID匹配
- **优先级排序**：数字越小优先级越高，确保高优先级规则先匹配
- **默认处理建议**：支持同意退款、驳回、人工审核三种建议
- **自动分派**：规则可指定分派客服，匹配后自动分派
- **启停状态**：规则可随时启用/禁用，禁用规则不参与匹配
- **CSV导入导出**：支持批量规则管理，导出格式可直接编辑后重新导入
- **冲突处理**：导入时自动检测并调整重复优先级，检测无效金额区间
- **并发控制**：使用版本号防止并发修改冲突
- **权限控制**：仅客服可管理规则，团长和商家仅能查看案件上的建议结果
- **审计日志**：规则创建、更新、删除、启用、禁用、命中、人工覆盖、导入、导出全记录
- **人工覆盖**：客服可人工覆盖系统建议，覆盖操作记录审计日志
- **数据持久化**：重启后规则配置、命中记录、分派结果、导出内容完全一致

### 规则匹配优先级说明
1. 系统按优先级从小到大遍历所有启用的规则
2. 每个规则可配置多个条件（售后类型、责任方、金额区间、商家）
3. 条件为空表示匹配任意值（通用规则）
4. 所有非空条件都匹配时，规则命中
5. 返回第一个命中的规则（优先级最高）

## 十、客服质检抽查模块

### 功能概述
客服可以从已完成退款、驳回或被撤销的售后案件里按日期、售后类型、责任方和处理人筛选，创建一批质检抽查单，每单快照保存案件关键信息、原裁决、证据链接、命中规则和导出记录摘要。质检员能给每笔标记通过、需复核、误判并填写原因，提交时用版本号防止同一抽查单被重复修改；已有结论被再次打开时只能追加复核记录，不能覆盖历史。模块需要有列表、详情、CSV 导入抽查清单、CSV 导出结果和操作日志，数据写进 SQLite，服务重启后抽查单、明细、复核轨迹和导出内容保持一致。只有客服可创建和处理，团长、商家无入口且接口拒绝。

### 核心特性
- **多维度筛选**：支持按日期范围、售后类型、责任方、处理人筛选已完成案件
- **数据快照**：创建抽查单时自动快照保存案件关键信息，确保历史数据不可变
- **乐观锁控制**：使用版本号防止并发修改冲突，旧版本提交被拒绝
- **复核轨迹**：已有结论的案件再次质检时只能追加复核记录，不能覆盖历史
- **状态流转**：待质检 → 通过/需复核/误判 → 追加复核记录
- **CSV导入导出**：支持导入抽查清单、导出质检结果，UTF-8 BOM编码处理中文
- **操作日志**：完整记录创建、质检、复核、导入、导出等所有操作
- **权限控制**：仅客服（cs角色）可访问和操作，团长、商家无权限
- **数据持久化**：SQLite存储，重启后抽查单、明细、复核轨迹、日志、导出内容完全一致

### 快照内容说明
每笔抽查明细快照包含：
- 案件基础信息：订单号、售后类型、商品名称、数量、退款金额、责任方、商家、问题描述
- 原裁决信息：原裁决（同意退款/驳回）、原裁决备注、原裁决人、原裁决时间
- 证据与规则：证据链接数组、命中规则、命中原因
- 导出记录：导出记录摘要
- 版本与状态：案件版本号、案件状态（已退款/已驳回/已撤销）

### 质检状态说明
| 状态 | 说明 |
|------|------|
| pending | 待质检 |
| passed | 通过 |
| needsReview | 需复核 |
| misjudged | 误判 |

## 十一、客服赔付承诺跟踪模块

### 功能概述
客服可以在售后详情页把已协商的补偿金额、优惠券、补寄商品或线下承诺登记成承诺单，并在独立页面查看待履约、已履约、逾期和取消状态。团长和商家只能看到与自己相关的只读摘要，越权接口会被拒绝。承诺单写入 SQLite，服务重启后状态、操作人、附件备注和取消原因都保留。支持从 CSV 批量导入承诺单并导出当前筛选结果；同一承诺被多人同时修改时用版本号提示冲突，不能静默覆盖。

### 核心特性
- **四种承诺类型**：现金赔付（cash）、优惠券（coupon）、补寄商品（reship）、线下承诺（offline）
- **四种状态管理**：待履约（pendingFulfillment）、已履约（fulfilled）、已逾期（overdue）、已取消（cancelled）
- **角色权限控制**：
  - 客服（cs）：可新建、编辑、取消、标记履约，查看全部承诺，导入导出CSV
  - 团长（leader）：只能查看与自己相关案件的承诺摘要（只读）
  - 商家（merchant）：只能查看与自己相关案件的承诺摘要（只读）
- **乐观锁并发控制**：使用版本号检测冲突，旧版本修改被拒绝，不静默覆盖
- **CSV导入导出**：支持批量导入承诺单，导出当前筛选结果，UTF-8 BOM编码处理中文
- **操作日志审计**：完整记录创建、编辑、履约、取消等所有操作，包含变更前后快照
- **数据持久化**：SQLite存储，重启后所有数据（状态、操作人、附件备注、取消原因）完全一致
- **逾期自动检测**：履约截止日期超过后自动标记为逾期状态

## 十二、失败链路验证

### 失败链路1：商家越权裁决

**预期错误**：`无权限执行此操作` / `PERMISSION_DENIED`

**复现步骤**：
1. 登录商家账号 `merchant1`
2. 找到一个状态为「客服仲裁」的案件
3. 尝试直接点击「同意退款」或「驳回」
4. 系统返回权限错误

**API验证**：
```bash
# 使用商家token调用客服裁决接口（会被拒绝）
curl -X POST http://localhost:3001/api/cases/{caseId}/action \
  -H "Authorization: Bearer {商家token}" \
  -H "Content-Type: application/json" \
  -d '{"action":"csRefund","version":2,"remark":"越权操作"}'
```

### 失败链路2：缺少凭证提交

**预期错误**：`请提供凭证URL` / `MISSING_EVIDENCE`

**复现步骤**：
1. 登录团长账号 `leader1`
2. 创建一个新的售后申请（状态为「待举证」）
3. 进入详情页，在「提交凭证」表单中**不填凭证URL**
4. 点击提交，系统返回错误

**API验证**：
```bash
# 提交凭证但缺少evidenceUrl
curl -X POST http://localhost:3001/api/cases/{caseId}/action \
  -H "Authorization: Bearer {团长token}" \
  -H "Content-Type: application/json" \
  -d '{"action":"submitEvidence","version":1,"remark":"缺少凭证"}'
```

### 失败链路3：旧版本重复处理

**预期错误**：`案件版本不匹配，请刷新后重试` / `VERSION_CONFLICT`

**复现步骤**：
1. 打开两个浏览器窗口，都登录团长账号
2. 进入同一个案件的详情页（版本号为v1）
3. 在窗口A中提交凭证，状态流转成功（版本变为v2）
4. 在窗口B中（仍显示v1）尝试提交凭证
5. 系统返回版本冲突错误

**API验证**：
```bash

  -H "Authorization: Bearer {团长token}" \
  -H "Content-Type: application/json" \
  -d '{"action":"submitEvidence","version":1,"evidenceUrl":"https://example.com/1.jpg","remark":"第一次提交"}'

# 第二次使用旧版本号提交（失败）
curl -X POST http://localhost:3001/api/cases/{caseId}/action \
  -H "Authorization: Bearer {团长token}" \
  -H "Content-Type: application/json" \
  -d '{"action":"submitEvidence","version":1,"evidenceUrl":"https://example.com/2.jpg","remark":"重复提交"}'
```

### Q: 启动后数据库文件在哪？
A: 在项目根目录的 `data/database.sqlite`

### Q: 如何重置数据？
A: 删除 `data/database.sqlite` 文件，重启服务即可重新初始化

### Q: 前端无法访问后端？
A: 确认后端服务已启动（端口3001），检查 vite.config.ts 中的代理配置

### Q: Token过期了怎么办？
A: 重新登录即可，Token有效期24小时

### Q: 演示账号登录失败（401错误）怎么办？
A: 这是bcryptjs版本兼容性问题。系统启动时会自动检测并修复密码哈希：
1. 停止服务（Ctrl+C）
2. 重新启动服务（`npm run dev`）
3. 观察启动日志，应该看到「修复了 X 个用户的密码哈希」
4. 重新登录即可

如果问题仍然存在，可以删除数据库文件重置：
```bash
# Windows PowerShell
Remove-Item data\database.sqlite, data\database.sqlite-wal, data\database.sqlite-shm -Force

# 然后重新启动服务
npm run dev
```
